# 资源加载问题分析报告

## 问题现象

即使外部目录的文件名正确，系统仍然使用JAR包内的前端文件，而不是外部目录的文件。

## 验证结果

### 1. MD5对比
- **外部目录index.html的MD5**: `202d5cb4b2644bb959fd2983b40eb4a8`
- **实际返回内容的MD5**: `944a1e1c329c11ede42f47897d78d023`
- **结果**: 文件不同，说明使用的是JAR包内的文件

### 2. 日志分析
```
2025-12-18 10:17:33.056 [main]  INFO --- o.s.b.a.web.servlet.WelcomePageHandlerMapping: 59 Adding welcome page: class path resource [static/index.html]
2025-12-18 10:17:33.145 [main]  INFO --- com.genersoft.iot.vmp.conf.WebMvcConfig: 45 使用外部静态资源目录: /opt/wvp/static/
```

**关键发现**：
- `WelcomePageHandlerMapping` 在 `WebMvcConfig` **之前**注册
- `WelcomePageHandlerMapping` 注册的是 `classpath:/static/index.html`（JAR包内）
- `WebMvcConfig` 虽然配置了外部目录，但已经晚了

## 问题根源

### Spring Boot的资源处理机制

1. **WelcomePageHandlerMapping（自动配置）**
   - Spring Boot自动配置，优先级：**最高**
   - 注册时机：在自定义`WebMvcConfigurer`之前
   - 处理路径：根路径 `/`
   - 查找位置：**只查找** `classpath:/static/index.html`
   - **不会查找外部目录**

2. **WebMvcConfig（自定义配置）**
   - 优先级：**较低**
   - 注册时机：在`WelcomePageHandlerMapping`之后
   - 处理路径：`/**`（所有路径）
   - 查找位置：外部目录 + JAR包内（后备）

### 调用流程

当访问 `http://localhost:18978/` 时：

```
1. 请求到达 Spring MVC DispatcherServlet
   ↓
2. HandlerMapping 查找处理器
   ↓
3. WelcomePageHandlerMapping 匹配根路径 "/"
   ✅ 找到处理器：classpath:/static/index.html
   ↓
4. 返回JAR包内的index.html
   ↓
5. WebMvcConfig的资源处理器（未执行，因为已经找到处理器）
```

### 为什么WebMvcConfig不生效？

1. **优先级问题**：
   - `WelcomePageHandlerMapping` 的优先级高于 `ResourceHandlerRegistry`
   - Spring Boot的自动配置在自定义配置之前执行

2. **路径匹配**：
   - `WelcomePageHandlerMapping` 专门处理根路径 `/`
   - 一旦匹配，就不会继续查找其他处理器

3. **资源查找范围**：
   - `WelcomePageHandlerMapping` **只查找** `classpath:` 资源
   - **不会查找** `file:` 协议的外部目录

## 解决方案

### 方案1：禁用WelcomePageHandlerMapping（推荐）

在`application-docker.yml`中添加：

```yaml
spring:
  web:
    resources:
      add-mappings: false
```

或者创建一个配置类禁用：

```java
@Configuration
public class DisableWelcomePageConfig {
    @Bean
    public WelcomePageHandlerMapping welcomePageHandlerMapping() {
        return null; // 禁用默认的欢迎页面
    }
}
```

### 方案2：配置WelcomePageHandlerMapping使用外部目录

创建一个自定义的`WelcomePageHandlerMapping`：

```java
@Configuration
@Order(Ordered.HIGHEST_PRECEDENCE)
public class CustomWelcomePageConfig {
    
    @Value("${web.static.path:/opt/wvp/static}")
    private String staticPath;
    
    @Bean
    public WelcomePageHandlerMapping welcomePageHandlerMapping(ApplicationContext applicationContext) {
        Resource resource = new FileSystemResource(staticPath + "/index.html");
        if (resource.exists()) {
            return new WelcomePageHandlerMapping(
                Collections.singletonMap("/", resource),
                applicationContext
            );
        }
        return null;
    }
}
```

### 方案3：使用ResourceHandlerRegistry的优先级

修改`WebMvcConfig`，使用更高的优先级：

```java
@Configuration
@Order(Ordered.HIGHEST_PRECEDENCE - 1) // 比WelcomePageHandlerMapping稍低
public class WebMvcConfig implements WebMvcConfigurer {
    // ... 现有代码
}
```

但这个方法可能不够，因为`WelcomePageHandlerMapping`是自动配置的。

### 方案4：使用Controller处理根路径

创建一个Controller来处理根路径：

```java
@Controller
public class IndexController {
    
    @Value("${web.static.path:/opt/wvp/static}")
    private String staticPath;
    
    @GetMapping("/")
    public String index() {
        Resource resource = new FileSystemResource(staticPath + "/index.html");
        if (resource.exists()) {
            return "forward:/index.html";
        }
        return "forward:/index.html"; // 后备到JAR包内
    }
}
```

## 推荐方案

**推荐使用方案1**：禁用`WelcomePageHandlerMapping`，让`WebMvcConfig`的资源处理器处理所有请求。

这样可以：
- ✅ 统一使用外部目录的资源
- ✅ 保持配置简单
- ✅ 不影响其他功能

## 验证步骤

修复后验证：
1. 修改外部目录的`index.html`（添加一个注释）
2. 访问 `http://localhost:18978/`
3. 检查返回的内容是否包含修改的注释
4. 对比MD5，确认使用外部目录的文件
